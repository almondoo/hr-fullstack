# SLA 目標 / 可用性設計メモ (Issue #19)

> **ステータス**: 設計検討フェーズ。SLA 数値は**ビジネス判断待ち**。
> クラウドプロバイダ選定 (GAP-01) 後に具体的なインフラ構成を確定する。
> `/healthz` `/readyz` エンドポイントは**実装済み** (下記 §2 参照)。

---

## 1. SLA 目標 (ビジネス判断待ち)

| レベル | 可用率 | 月間許容ダウンタイム | 年間許容ダウンタイム | 想定コスト増 |
|--------|--------|---------------------|---------------------|-------------|
| **99.0%** | 月 7.2 時間 / 年 87.6 時間 | シングル AZ で達成可能 | 低 |
| **99.5%** | 月 3.6 時間 / 年 43.8 時間 | マルチ AZ + 自動フェイルオーバー | 中 |
| **99.9%** | 月 43.8 分 / 年 8.7 時間 | マルチ AZ + 複数インスタンス + 自動スケーリング | 中〜高 |
| **99.95%** | 月 21.9 分 / 年 4.4 時間 | マルチリージョン / アクティブ-アクティブ | 高 |

> **推奨初期目標**: **99.9%** — SaaS 外販の最低ラインとして業界標準。
> ただし、SLA 数値は顧客契約・ペナルティ条件と一体で決定すべきビジネス判断。
> 事業責任者と合意の上で確定すること。

**SLA の計測方法** (確定後に定義):
- 対象: API エンドポイントへの HTTP レスポンス (5xx エラー率)
- 除外: 計画メンテナンス時間 (事前通知済み)
- 計測ツール: 外部死活監視 (Pingdom / UptimeRobot / CloudWatch Synthetic 等) — TBD

## 2. 実装済み — ヘルスチェックエンドポイント

`/healthz` および `/readyz` は実装済み (backend/internal/server/ 参照)。

| エンドポイント | 用途 | 確認内容 |
|---------------|------|---------|
| `GET /healthz` | 起動確認 (Liveness) | プロセスが生存しているか |
| `GET /readyz` | トラフィック受け入れ可否 (Readiness) | DB 接続・依存サービスの確認 |

コンテナオーケストレーター (ECS / Cloud Run / Container Apps) のヘルスチェック設定で
これらのエンドポイントを使用する (`infra/modules/container/main.tf` 参照)。

## 3. 冗長化 / ロードバランサ設計

### 3-1. アプリケーション層

| 構成 | 最小インスタンス数 | 対応 SLA | 備考 |
|------|-----------------|---------|------|
| シングルインスタンス | 1 | ~99.0% | 本番非推奨 |
| マルチ AZ (同リージョン) | 2 (AZ ごとに 1) | 99.5〜99.9% | **推奨初期構成** |
| マルチリージョン | 4+ | 99.95%+ | SLA 要件が高い場合 |

**初期推奨**: 最小 **2 インスタンス** を別 AZ に配置 + ロードバランサ自動分散。
`infra/modules/container/main.tf` の `min_instances = 2` が対応するデフォルト値。

### 3-2. データベース層

| 構成 | フェイルオーバー時間 | 対応 SLA |
|------|-------------------|---------|
| シングル AZ | 手動復旧 (数時間) | ~99.0% |
| Multi-AZ (自動フェイルオーバー) | 30〜120 秒 | 99.5〜99.9% |
| クロスリージョンレプリカ | 数分 (手動昇格) | 99.95%+ |

**初期推奨**: **Multi-AZ** を有効化。`infra/modules/database/main.tf` で設定。

### 3-3. ロードバランサ

| プロバイダ | サービス | 特徴 |
|-----------|---------|------|
| AWS | ALB (Application Load Balancer) | HTTP/2・WebSocket 対応、WAF 統合 |
| GCP | Cloud Load Balancing | グローバル / リージョナル選択可、Cloud Armor 統合 |
| Azure | Application Gateway / Front Door | WAF 統合、グローバル負荷分散 |

ロードバランサは `/healthz` で Liveness チェック、`/readyz` で Readiness チェックを実施し、
異常なインスタンスへのトラフィックを自動的に切り離す。

## 5. 計画停止 / メンテナンス運用

**手順 (案)**:

1. **事前通知**: メンテナンス開始の 72 時間前までに顧客へメール通知 + ステータスページ更新
2. **ローリングデプロイ**: 新バージョンを順次デプロイし、ダウンタイムをゼロに近づける
   - ECS / Cloud Run / Container Apps はいずれもローリングアップデートを標準サポート
3. **DB マイグレーション**: 後方互換マイグレーションを先行実施し、アプリデプロイと分離する
   - `backend/db/migrations/` の goose マイグレーションは後方互換を原則とする
4. **ロールバック**: 異常検知時はタスク定義の前バージョンに即時ロールバック
5. **完了通知**: メンテナンス完了をステータスページで更新

**ステータスページ**: TBD (Statuspage / BetterStack / 自前等) — クラウド選定後に決定。

## 6. オートヒーリング設定 (IaC 実装時)

コンテナオーケストレーターのヘルスチェック設定 (`infra/modules/container/main.tf` 実装時):

```hcl
# 例 (AWS ECS の場合 — TODO(#9): プロバイダ確定後に実装)
# health_check {
#   command     = ["CMD-SHELL", "curl -f http://localhost:8080/healthz || exit 1"]
#   interval    = 30
#   timeout     = 5
#   retries     = 3
#   start_period = 60
# }
```

**Liveness 失敗**: コンテナを再起動 (プロセス異常・デッドロック対策)
**Readiness 失敗**: ロードバランサからトラフィックを切り離す (DB 接続異常時)

- 分散トレーシング (OpenTelemetry)
- アラート設定 (エラー率・レイテンシ・可用率)
- オンコール体制
