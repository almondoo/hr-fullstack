# 02 / フェーズ別プレイブック（進め方とゲート条件）

各フェーズの **手順・観点・成果物・ゲート（承認）条件** を定義する。
Claude はマスターワークフロー（`00`）の状態機械に従い、本書の手順を実行する。

---

## Phase 1 — 要件定義

### 手順（順に実施。各ステップ完了で WORKFLOW STATE を更新）

1. **ステークホルダー & ペルソナ定義**
   - 利用者：人事/労務担当、経営者、現場マネージャー、一般従業員、（社外）社労士。
   - ペルソナごとの主要ジョブ（JTBD）と痛点を1〜2行で整理。
2. **業務フロー / ユースケース抽出**
   - 領域ごとの主要シナリオ（例：入社手続き、勤怠締め、評価サイクル、選考〜内定）。
   - 各ユースケースに **アクター・前提・本流・例外** を簡潔に。
3. **機能要件の起票**
   - `01_domain_knowledge.md` の A〜D を**チェックリストとして全項目走査**。
   - 採用/除外を判断し、採用は要件IDを引用、除外は理由を記録。
   - 不足機能は新規IDを採番して追加。
4. **非機能要件（NFR）**
   - `01 > F` を走査し、SLA・性能・セキュリティ・マルチテナント方針の**目標値の素案**を置く（仮説台帳）。
5. **法令・コンプラ要件**
   - `01 > E` を走査。該当要件に「**要法務/社労士レビュー**」フラグを付与。
6. **スコープ & MVP 定義**
   - 全要件を **MVP / Fast-Follow / Future** に三分割。
   - MVPは「最初の有料顧客が業務を回せる最小」を基準にする。
7. **優先度付け**
   - MoSCoW＋必要に応じ **RICE**（Reach×Impact×Confidence÷Effort）でMVP内も順位付け。
8. **受入基準 & トレーサビリティ**
   - 主要要件に Given/When/Then 形式の受入基準。
   - 要件ID ↔ ユースケース ↔ （後続）設計IDの対応表を用意。

### ゲート1 チェックリスト（すべて満たしたら承認を求める）
- [ ] 3領域＋共通基盤の機能要件が網羅され、採否理由がある
- [ ] NFR・法令要件が起票され、専門家確認フラグが付いている
- [ ] MVP / Fast-Follow / Future が明確に分かれている
- [ ] MVP内の優先度と受入基準がある
- [ ] 仮説台帳・決定台帳が更新されている

→ 要約提示し、ユーザーに **`承認` / `修正` / `戻る`** を要求。

## Phase 2 — アーキテクチャ / 詳細設計

### 手順

1. **アーキテクチャ方式選定**
   - 候補：モジュラモノリス（推奨初期）／マイクロサービス／その中間。
   - SMB向けSaaS・小規模チーム前提では **モジュラモノリス＋将来分割可能な境界** を基本線として比較提示。
2. **マルチテナント分離方式の決定**（`NFR-001`）
   - 候補比較：共有DB＋テナントID（行レベル分離）／スキーマ分離／DB分離。
   - 機微情報（マイナンバー）の**分離強化**（別ストア/トークン化）を併せて設計。
   - 決定理由・トレードオフ（コスト/分離強度/運用）を決定台帳へ。
3. **ドメイン / データモデル**
   - 主要エンティティとリレーションを定義（最低限：Tenant, Employee, Org/Department, EmploymentContract, Attendance, LeaveBalance, PayrollLink, Applicant, JobPosting, SelectionStage, Onboarding Task, Goal/Review, OneOnOne, Skill, Survey, AuditLog, MyNumber(分離) など）。
   - 各領域の境界（Bounded Context）を示し、共有マスタ（従業員/組織）を中心に据える。
4. **API / インターフェース設計**
   - 方式（REST/GraphQL）、リソース設計、認証認可、ページング、Webhook、外部公開APIの方針。
5. **権限モデル（RBAC/ABAC）**（`NFR-002`）
   - ロール定義・操作権限・**項目レベル/テナント境界**の表現、職務分掌。
6. **セキュリティ & プライバシー設計**（`01 > E/F`）
   - 認証(MFA/SSO)、暗号化、監査ログ、秘密管理、脆弱性対策、データ保持/削除、Privacy by Design。
   - 認証取得（ISMS/Pマーク）を見据えた統制・証跡。
7. **外部連携設計**（`01 > G`）
   - e-Gov/マイナポータルAPI、給与、SSO、チャット等の連携方式・エラー/再送・データ整合。
8. **画面 / 情報設計（IA）**
   - 主要画面一覧、ナビゲーション、ロール別ビュー、従業員モバイルWeb。
9. **技術スタック選定**
   - 言語/FW、DB、インフラ（クラウド/コンテナ）、認証基盤、ジョブ基盤、監視を
     **要件と体制から逆算**して推奨＋代替案。制約があればそれを優先。

### ゲート2 チェックリスト
- [ ] アーキ方式とマルチテナント分離方式が**理由つき**で決定されている
- [ ] 主要エンティティのデータモデルと境界が定義されている
- [ ] 権限モデルとテナント越境防止が設計されている
- [ ] セキュリティ/プライバシー/監査が要件を満たす
- [ ] 外部連携方式と技術スタックが決定（または推奨提示）されている
- [ ] 要件ID → 設計要素のトレーサビリティがある

→ 要約提示し **`承認` / `修正` / `戻る`**。

## Phase 3 — 実装計画

### 手順

1. **ロードマップ / マイルストン**
   - フェーズ（例：基盤→労務MVP→ATS→タレマネ）とおおよその時間軸。
2. **MVPスコープ確定**
   - Phase1のMVPを実装単位に落とす（リリース可能な最小の縦切り）。
3. **エピック → ユーザーストーリー分解**
   - 要件IDからエピックを作り、ストーリーに分解。受入基準を引き継ぐ。
4. **体制 & 見積**
   - 役割（PO/アーキ/BE/FE/QA/SRE/デザイン）、概算工数（T-shirt or ストーリーポイント）、前提を明記。
   - 不確実性は**レンジ**で示し、仮説を添える（過度に精緻化しない）。
5. **環境 & CI/CD**
   - 環境分離（dev/stg/prod）、IaC、CI/CD、シークレット管理、リリース戦略。
6. **テスト戦略**
   - 単体/結合/E2E、セキュリティテスト、**法令計算（勤怠/割増/有給）の検証**、テストデータ（個人情報を使わない）。
7. **リリース & データ移行**
   - 顧客オンボーディング、初期データ移行（既存HRデータ取込）、段階リリース/フィーチャーフラグ。
8. **リスク & 対応**
   - 技術/法令/スケジュール/セキュリティのリスクを **発生度×影響度** で整理し対応策。
9. **コスト**
   - インフラ/外部サービス/認証取得の概算ランニングコスト観点。
10. **成功指標（KPI）**
   - プロダクトKPI（アクティベーション/継続/NRR）と品質KPI（障害/重大バグ）。

### ゲート3 チェックリスト
- [ ] ロードマップとMVPの実装スコープが定義されている
- [ ] エピック/ストーリーに分解され受入基準がある
- [ ] 体制・見積（レンジ・前提つき）がある
- [ ] テスト戦略に**法令計算の検証**が含まれる
- [ ] リスクと対応、リリース/移行方針がある

→ 要約提示し **`承認` / `修正` / `戻る`**。承認後 Phase 4（ハンドオフ）へ。

## 共通：ゲートでのユーザー操作

| 入力 | 挙動 |
|---|---|
| `承認` | 次フェーズへ進む。決定を決定台帳に確定。 |
| `修正: <内容>` | 当該フェーズ内で修正し、再度ゲート提示。 |
| `戻る: <Phase>` | 指定フェーズへ戻り、**下流（設計/計画）への影響**を洗い出して提示。 |
| `深掘り: <項目>` | 該当項目を詳細化（必要なら並列タスク/検索で裏取り）。 |
