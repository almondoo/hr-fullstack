## 2. バックアップ戦略

### 2-1. PostgreSQL 自動バックアップ (マネージドサービス)

クラウドプロバイダのマネージド PostgreSQL を使用する場合、以下が自動的に提供される。
IaC (`infra/modules/database/main.tf`) で有効化する。

| プロバイダ | サービス | PITR 対応 | 設定箇所 |
|-----------|---------|-----------|---------|
| AWS | RDS for PostgreSQL | 最大 35 日 | `backup_retention_period` |
| GCP | Cloud SQL | 最大 365 日 | `backup_configuration.point_in_time_recovery_enabled` |
| Azure | Flexible Server | 最大 35 日 | `backup_retention_days` + `geo_redundant_backup_enabled` |

**最低設定**:
- 自動バックアップ: 有効
- PITR (Point-in-Time Recovery): 有効
- バックアップウィンドウ: 低トラフィック時間帯に設定
- 暗号化: KMS CMK で暗号化 (infra/modules/secrets/ 参照)

### 2-2. WAL アーカイブ (セルフマネージドまたは補完)

マネージドサービスの PITR が RPO を満たさない場合や、長期アーカイブが必要な場合:

- `pgBackRest` または `WAL-G` を使用したオブジェクトストレージへの WAL アーカイブ
- バックアップ先: S3 / GCS / Azure Blob Storage (クラウド選定後に確定)

### 2-3. アプリケーション設定・シークレット

## 3. リストア手順 (概要)

> **注意**: 実際のリストア手順は運用チームが定期的に検証し、ランブックとして保管する。
> 以下は設計段階の概要であり、実際の操作コマンドはプロバイダ確定後に具体化する。

### 3-1. データベースリストア

1. **障害確認**: DB インスタンスの状態・エラーログを確認
2. **PITR リストア**: シークレットマネージャから接続情報を取得し、マネージドコンソールまたは IaC で PITR を指定
3. **接続文字列更新**: リストア先エンドポイントをシークレットマネージャで更新
4. **アプリ再起動**: コンテナを再起動して新しい接続文字列を反映
5. **整合性確認**: テナントデータの件数・最終更新日時を確認

### 3-2. 暗号化データのリカバリ

列暗号 (AES-256-GCM) で暗号化されたデータのリカバリには、バックアップ時点の鍵が必要。

- **鍵ローテーション前のデータ**: 旧鍵を使ってのみ復号可能。旧鍵はシークレットマネージャのバージョン履歴に保持する。
- **鍵紛失時**: 対応する列のデータは復号不可能。鍵は絶対に削除しない。
- **手順**: `crypto.KeyProvider` の旧バージョン鍵を使った再暗号化ツールは #10 完了後に実装する。

### 3-3. 復旧テスト

**定期実施 (推奨: 四半期に一度)**:
- 別環境へのリストアで手順を検証
- RTO 目標内に完了できることを確認
- テスト結果を記録・レビュー

## 4. 鍵ローテーション手順 (概要)

FIELD_ENCRYPTION_KEY のローテーションは**データ損失リスクが高い**操作であり、慎重に実施する。

1. 新しい鍵をシークレットマネージャに登録 (旧鍵は削除しない)
2. バッチプロセスで全暗号列を旧鍵で復号 → 新鍵で再暗号化
   - TODO(#10): 再暗号化バッチは KMS 実装完了後に実装
3. 再暗号化完了後、アプリを新鍵で再起動
4. 旧鍵は最低 30 日間保持してから廃棄 (ロールバック対応)

## 5. リージョン冗長化 (検討事項)

| 方式 | 説明 | コスト | 複雑度 |
|------|------|--------|--------|
| マルチ AZ (同リージョン) | DB は Multi-AZ、コンテナは複数 AZ に分散 | 中 | 低 |
| リードレプリカ (読み取り分散) | 参照系 SQL をレプリカに向けてプライマリ負荷軽減 | 中 | 中 |
| クロスリージョンレプリカ | DR 用途。別リージョンに非同期レプリカを維持 | 高 | 高 |
| バックアップのみ (コールドスタンバイ) | 別リージョンにバックアップを保存し障害時に手動復旧 | 低 | 低 (RTO 長) |

> **推奨**: 初期は**マルチ AZ + バックアップのリージョン間レプリケーション**で開始し、
> SLA 要件と利用規模に応じてクロスリージョンレプリカを検討する。
> 最終決定はビジネス要件・コスト試算を経て事業責任者が判断する。
